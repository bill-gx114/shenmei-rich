## 0. 30 秒电梯陈述

**审美日课** 是一个「每天精读一幅名画」的极简博物馆产品：每天系统自动选一幅 textbook 级别的世界名作，配上 AI 生成的策展导览（看点热区、三档语气的语音讲解、墙签、三道观察题、词条），用户答题后，他选用的关键词会自动沉淀成「审美词典」和「正在形成的判断」——把碎片化的观看，变成可累积的审美判断力。

我一个人完成了**从产品定义、视觉落地、全栈实现到上线运维**的全过程：用 Vite+React 重建了一套 Claude 生成的设计稿，用 Supabase 做多租户后端，接 DeepSeek 做内容生产、接微软神经语音做导览发声，最后部署到 Vercel + Supabase 的全免费技术栈，**单月成本 0 元，支撑 10 人朋友圈日常使用**，且内容能**零人工自动日更**。

---

## 1. 为什么做这个（产品出发点）

- **真问题**：很多人想「提升审美」，但缺的不是图，是**判断的脚手架**——看一幅画时不知道看什么、看完留不下东西。市面上要么是图库（只供给，不引导），要么是课程（太重，不日常）。
- **产品主张**：把「审美」拆成可以**每天 5 分钟、可累积、可回看**的微习惯。核心不是「看了多少画」，而是「你逐渐稳定下来的判断维度」。
- **形态选择**：博物馆隐喻（今日展厅 / 馆藏 / 观察手账三个 tab），克制的暗厅美学，降低「学习」的压迫感，强调「逛」与「沉淀」。

> 面试可讲点：这体现了**从模糊需求到可度量微习惯的产品拆解**——把"提升审美"这种不可证伪的目标，落成"连续打卡 / 审美词典词数 / 正在形成的判断"这类可见的累积指标。

## 2. 核心体验（产品功能地图）

| 模块 | 用户能做什么 | 背后的设计意图 |
|---|---|---|

| **录入页（Admin）** | 管理员手动录入作品 + AI 一键起稿 | 长期可运营的后门 |

| **公开审美主页（/u/handle）** | 一键把审美肖像 + 词典 + 馆藏精选做成可分享名片；移动端原生分享 + 动态生成的预览海报 | 把个人积累变成可传播资产，在 10 人小图里制造社交动力 |

### 技术栈一句话清单

- **前端**：Vite + React 18 + TypeScript + react-router；localStorage 持久化用户偏好（Tweaks）
- **后端**：Supabase —— Postgres（业务数据）+ Row Level Security（多租户隔离）+ Auth（邮箱密码 / GitHub OAuth）+ Storage（画作图、语音缓存）
- **Serverless**：Vercel Functions（含 cron 定时任务）
- **AI 内容生产**：DeepSeek `deepseek-chat`，`response_format: json_object` 强约束结构化输出
- **语音合成**：微软 Edge 神经语音（晓晓/云希/晓伊），WebSocket 协议手写代理 + Supabase Storage 缓存
- **图片来源**：Wikipedia REST Summary API（公版图，零版权风险，不硬编码 URL）
- **部署**：Vercel（前端+函数）+ Supabase（托管后端），全免费档

## 4. 关键产品/技术决策与权衡

这部分是面试最值得讲的——**每个选择背后的取舍逻辑**。

### 4.1 多用户 vs 单用户
选了**多用户 + 每人独立馆藏**。代价是要做鉴权、RLS、数据隔离，复杂度上去了；收益是产品能给朋友用、能看到留存差异。**今日作品是全局共享**（一张画所有人看同一张），但**答题/收藏/词典是每用户私有**——这个"公共内容 + 私有进度"的混合模型，是刻意设计的：既保证内容供给统一，又保证个人进度感。

### 4.2 内容生产：人工 vs AI 自动
日更产品最大的死穴是**内容枯竭**。我设计了三层防枯竭：
1. 代码里内置 30 幅 textbook 名作种子库
2. `daily-curator` 每天按序轮换选一幅，调 DeepSeek 现写策展包
3. `extend-seed` 每月让 DeepSeek 推荐 10 幅新名作、用 Wikipedia 校验有图后入库

结果：**内容池永不枯竭、零人工、每天自动产出**。这是把"AI 当成内容供给侧的生产力"而不是"聊天玩具"的典型用法。

### 4.3 语音：免费系统 TTS vs 神经语音
先用浏览器自带 Web Speech API（零成本、离线），但音质太"塑料"。最终上微软 Edge 神经语音（音质接近真人），用 **Supabase Storage 缓存每一句**——同一句第二次播放直接命中缓存、零延迟、零重复合成开销。两种引擎都保留，用户可在 Tweaks 里切换。**这是"体验 vs 成本"的经典权衡，用缓存把边际成本压到趋近 0。**

### 4.4 登录门槛：先登录 vs 先体验
最初整个 app 被登录页挡住，首屏即表单。改成**匿名先逛**——今日展厅、导览、馆藏都能看，只有"收入手账/收藏/录入"这类写操作才弹登录，并带 `returnTo` 登录后回到原地。**降低首屏流失，把鉴权摩擦推迟到用户已经产生兴趣之后。**

### 4.5 部署平台：Netlify → Vercel 迁移
开发期在 Netlify，因为高频 push 触发构建，21 次部署吃掉 315 个免费 credit 直接限额、网站打不开。迁到 Vercel（部署次数不计费）。**迁移时刻意放在新文件夹 `web-vercel/`、保留原 `web/` 做灾备**，把 Netlify Functions 平移成 Vercel Functions，前端零改动（两边都用 `/api/*` 路径）。

### 4.6 继续堆功能 vs 先装仪表盘 ⭐ 数据驱动
做到一定阶段，手上攒了一串想做的功能（社交对照、日更提醒、洞察反向选画）。但我**先停下来做了埋点**——因为产品此前零埋点，朋友到底回不回来、哪道题被跳过、改造后的词典有没有人点，全靠猜。**先装仪表盘，再决定往哪飞**：埋点最便宜、最不打扰现有体验，且不做它，后面任何功能上了也无法判断有没有用。落地为一张 write-only 的 `events` 表 + 一个 fire-and-forget 的 `track()`（非阻塞、出错静默，绝不拖累 UX），埋了打开 / 看画 / 答题 / 跳题 / 功能使用 / 词典点击等 11 个事件，配一组现成 SQL（留存、漏斗、跳题率）。**这是我从策略 PM 带过来、最不愿丢的肌肉：用数据立论，而不是拍脑袋排优先级。**

> 真实闭环：埋点上线第一天就读出"会话几乎全部止步于看画，零答题、零登录"，于是当晚就做了"匿名可答题、答案跨登录无损保留"降门槛；再下一步，按数据指出的"缺乏传播力"做了公开分享（§4.7）。**埋点 → 读数 → 改产品**，三天跑了一个完整循环。

### 4.7 分享能力：优雅与隐私的平衡 ⭐
数据显示产品缺乏传播力，而它的独特资产恰恰是"一小群朋友看同一幅画"。于是做了**公开审美主页** `/u/<handle>`。三个权衡：
- **隐私**：公开是 opt-in 的，且**只暴露白名单字段**（审美肖像 / 词典 / 统计 / 馆藏精选），私人观察笔记永不公开。吸取早期"视图泄漏"教训，公开数据由服务端**显式拼装**，绝不走 RLS 暴露的视图。
- **移动端优先**：分享用 `navigator.share()` 调起系统面板（微信 / iMessage），桌面端降级复制链接——一套代码两种最优体验。
- **预览即门面**：SPA 默认抓不到动态 OG，所以做了一个边缘函数：`/u/<handle>` 返回注入了 per-profile OG 标签的页面，并用 `@vercel/og` **动态生成一张深底暖金的分享海报**（名字 + AI 肖像金句 + 统计）。链接贴进微信弹出的那张卡片，才是传播的真正入口。

## 5. AI 在产品里的四种用法（AI PM 重点）

我有意识地把 AI 用在四个不同层面，而不是"塞个聊天框"：

1. **结构化内容生产**（DeepSeek）
   - 输入：作品标题 + 作者 + 一句话感觉
   - 输出：严格 JSON 的「策展包」——墙签、3 个看点热区（带坐标）、**三档语气各一套语音脚本**、3 道观察题、3-5 个词条、一段第一人称策展人留言
   - 关键工程：用 `response_format: json_object` 强约束；prompt 里写死风格基线（克制、术语准、永远讲"为什么这件作品成立"）；对 AI 偶尔漏字段做容错降级；`max_tokens` 拉满防截断（见踩坑 2）
2. **内容池自维护**（DeepSeek + 校验闭环）
   - 让 AI 推荐新名作，但**不盲信** —— 每条候选都用 Wikipedia 搜索 API 多语种多查询验证"真的存在且有图"，验不过就拒。**AI 负责创意，规则负责把关**，这是降低 AI 幻觉风险的产品化手段。
   - 每周自动扩充，使选品池增速持续高于日更消耗，内容永不重复。
3. **AI 作为洞察引擎**（最能体现 AI PM 价值）⭐
   - 「正在形成的判断」最初是模板（"X 出现 3 次"），价值很弱。升级为：把用户**真实的答题记录 + 关键词频次**喂给 DeepSeek，写出一段第二人称的「审美肖像」+ 2-4 条具体倾向。
   - 实际产出举例（真实生成）："你被两种极端的视觉张力吸引：东方版画中翻腾的巨浪与赤铜山体，西方油画里旋转的星夜与模糊的微笑……你正在形成一种偏好——放弃轮廓线、用色块构建形态，并放大局部细节以打破平衡。"
   - **同样的数据，模板只能说"X 出现 3 次"，大模型能说出"你正在形成一种偏好"并跨作品归纳出连贯人格。** 这是把 AI 从"内容生成器"用成"洞察引擎"的质变。
   - 工程：服务端校验 JWT → 读该用户数据 → 生成 → 缓存（`user_insights` 表）。**只在用户答了新题（entry_count 变化）时才重新生成**，否则命中缓存，把 LLM 调用压到最低；不足 3 条记录时不调用、给引导文案。
4. **语音合成**（微软神经 TTS）
   - 三档语气映射三个不同神经音色，让"策展人语气"这个产品概念真正可听。

> 面试可讲点：**AI 产品不是"接个大模型"，而是设计"AI 生成 → 规则校验 → 缓存复用"的可靠管道**。我在这个项目里反复体现了"用确定性的工程约束去驯化不确定的模型输出"——而第 3 点（洞察引擎）更进一步：**AI 真正的杠杆不在替人写内容，而在从用户行为里读出连人自己都没明说的模式**。

## 6. 踩坑实录（War Stories —— 面试最爱听）

### 坑 1：微软 Edge 神经语音，6 轮才打通 ⭐ 最硬核
想白嫖微软 Edge 浏览器内部的神经 TTS，踩了一条长链：
1. Vercel Edge runtime 原生 WebSocket **不能设自定义 header** → 微软拒连
2. 换 Node runtime 但用 Web Fetch 签名 → Vercel 不识别，函数**冷启动挂死 15 秒**
3. 改回 legacy `(req,res)` 签名 → 通了，但微软返回 **403**
4. 403 是因为微软 2024 年加了 `Sec-MS-GEC` 反爬签名（时间戳 hash）→ 补算
5. 补了还是 403 → 发现是 **JS 浮点精度**问题：ticks ≈ 1.3×10¹⁷ 超过 `Number` 安全整数（2⁵³），`Math.floor` 静默丢精度导致 hash 永远错 → 改用 **BigInt** 全程整数运算
6. 再对齐 header（Pragma/Cache-Control/User-Agent 版本号）→ **终于 200，吐出真 mp3**

**教训**：第三方"灰色"能力的逆向集成，要有"逐层二分定位"的耐心；浮点精度在密码学/签名场景是隐形杀手。

### 坑 2：LLM 结构化输出被静默截断 ⭐ AI 工程化
某天日更的作品只有看点热区，**语音导览和题目全空了**。排查：works 行在、hotspots 有 3 条，但 audio_lines / questions / vocabulary 都是 0 行。根因是 **DeepSeek 调用没设 `max_tokens`，默认 4096 装不下"三档语气各一套语音脚本"这种超大响应**；而 `response_format: json_object` 模式在 token 耗尽时**仍返回语法合法的 JSON，只是提前闭合了对象**——prompt 里靠前的字段（shortLabel/curatorNote/hotspots）保住了，靠后的（audioLines/questions/vocabulary）被整段丢弃，且**不报任何错**。修复：`max_tokens: 8192`（模型输出上限）+ 对"内容为空"也主动记一条 `childError`，不再静默。

**教训**：① LLM 的"合法 JSON"≠"完整 JSON"，结构化输出必须考虑 token 预算与截断；② 把**最关键的字段放在 prompt 靠前**的位置（截断时先保住核心）；③ 对模型产出做**完整性校验**（空数组也是异常信号），别让"看起来成功的 200"掩盖内容缺失。这是典型的"模型不确定性需要工程兜底"的案例。

### 坑 3：Postgres 视图泄漏了别人的数据 ⭐ 安全
朋友登录后，"正在形成的判断"里出现了**别人的关键词**。排查发现：`v_user_constellation` 视图默认是 `SECURITY DEFINER` 语义——以视图所有者（postgres）权限查底层表，**绕过了 RLS**。修复：`security_invoker = true` 让视图按调用者权限跑 + 前端显式 `eq('owner_id')` 双保险。

**教训**：RLS 不会自动穿透视图。多租户系统里，**视图是 RLS 的盲区**，必须单独处理。

### 坑 4：连续打卡天数不准 ⭐ 数据语义
用户重答旧作品后，连续打卡天数变少了。原因：`notebook_entries` 按 `(user, work)` upsert，重答会把 `saved_at` 覆盖到当天，于是用 `saved_at` 算 streak 时旧日期"消失"。修复：改用作品的 `exhibited_on`（展出日期）算 streak——**"你观察过 X 日的展品"是稳定事实，跟你哪天点保存无关**。

**教训**：统计指标要选**语义稳定的时间锚点**，别用会被业务操作覆盖的字段。

### 坑 5：React Hooks 顺序违规导致白屏
给作品详情页加"收藏"时，把 `useNotebookEntry()` 放在了 `if (loading) return` 之后 → 加载态少调一个 hook → React 报错白屏。**教训**：所有 hook 必须在任何 early-return 之前无条件调用。

### 坑 6：AI 给的中文 Wikipedia slug 拉不到图
DeepSeek 给"创世纪""吻"这类中文 slug，但中文维基这些是**消歧义页**（没主图）。修复：多语种（zh+en）多查询（标题/作者罗马字）轮询 Wikipedia 搜索 API，挨个探主图。**教训**：AI 给的"看起来对"的标识符，落地前必须用真实接口验证。

### 坑 7：分享海报里的中文渲染不出来
`@vercel/og`（Satori）只内置拉丁字体，而完整中文字体好几 MB，塞不进边缘函数。修复：按要绘制的**确切文字**去 Google Fonts 拉一个**子集字体**（`css2?...&text=要画的字`），且用一个老旧 UA 骗 Google 返回 Satori 能解析的 TTF（默认会给 woff2）。**教训**：边缘渲染 CJK 是个真实约束，按需子集化是标准解法。

### 其它小坑
- 图片 `object-fit: cover` 把竖构图的画裁掉头/脚 → 改 `contain`
- Vite 只暴露 `VITE_` 前缀的环境变量，前端拿不到 `ADMIN_EMAIL` → 加 `VITE_ADMIN_EMAIL`
- Vercel 选 Root Directory 的 UI 有缓存 bug，新目录不显示 → Settings 里手输路径绕过

## 7. 数据模型与多租户安全

核心表（每张 enable RLS，策略 `owner_id = auth.uid()`）：

| 表 | 说明 |
|---|---|
| `works` | 作品（owner_id 为 null = 全局日更；非 null = 个人录入） |
| `hotspots` / `audio_lines` / `questions` / `vocabulary` | 作品的子内容（1:N） |
| `notebook_entries` | 用户答题（按 owner+work upsert） |
| `keyword_uses` | 关键词命中（驱动审美词典） |
| `user_pins` | 收藏（每用户独立，因为 RLS 不让用户改全局作品的字段） |
| `seed_works` | 名作种子库（AI 月度扩充） |

**安全设计亮点**：
- 全局作品任何人可读、只有 service_role（cron）能写
- 个人数据严格 `owner_id` 隔离
- 收藏单独建关系表而非在 works 上加字段——因为全局作品用户无权 UPDATE
- 视图显式 `security_invoker` 防 RLS 绕过

## 8. 可持续运营设计

- **零人工日更**：cron 每天北京 00:00 自动发布全局作品，DeepSeek 现场生成全部导览内容
- **内容池自增长**：每月 1 号 AI 推荐 10 幅新名作 + Wikipedia 校验入库，种子库永不枯竭
- **语音缓存**：每句导览首次合成后存 Storage，全站复用
- **幂等性**：daily-curator 同一天重复调用不会重复发布（先查 owner_id is null + 当天）

## 10. 复盘：作为（准）AI 产品经理，我证明了什么

1. **完整的产品闭环能力**：一个人把"模糊需求 → 产品定义 → 视觉 → 全栈 → 上线 → 运维 → 迭代"跑通，理解每一环的约束。
2. **AI 工程的产品化思维**：不是"调个 API"，而是设计"生成→校验→缓存→降级"的可靠管道，懂得用确定性工程约束去管理模型的不确定性。
3. **务实的技术权衡**：免费栈选型、成本边际优化、灰色能力的合规边界判断——都是 PM 跟工程/法务对话时的硬通货。
4. **对数据与安全的敏感**：RLS 多租户隔离、视图泄漏、指标语义稳定性——这些"看不见的正确"是 toC 产品的底线。
5. **快速排障与定位**：6 轮调通神经语音、视图泄漏溯源，体现了"假设→验证→收敛"的工程方法论。
6. **数据驱动而非拍脑袋**：在"继续堆功能"和"先装仪表盘"之间，我选择先埋点（§4.6）——用留存、漏斗、跳题率来决定下一步做什么，把策略 PM 的"用数据立论"习惯带进了 AI 产品。

> 一句话面试结尾：**这个项目让我把"产品想法"真正变成了"在线上跑、有人用、能自我维持的系统"——我理解了 AI 产品从 idea 到 production 之间，每一道工程与权衡的缝。**
